Engineering Gödel

A humble journey into building a deterministic UI engine. We focused on metrics that matter: binary size, latency, and developer sanity.

Architecture

What is CGO (and why we removed it)?

In the Go world, CGO is a tool that allows Go code to call C code. For most GPU frameworks, this "bridge" is mandatory because graphics drivers were traditionally written for C.

The "Bridge" Analogy

Imaging building a house where half the workers speak English (Go) and the other half speak Latin (C). Every time they need to talk, they have to stop, find a translator, and explain the plan. This "bridge" causes latency (delay) and makes the building process (the app) slower and more complex.

Gödel is "Zero-CGO." We taught Go how to speak directly to the GPU's native language. There's no bridge, no translator, and no delay. This is why Gödel apps start instantly and feel so much snappier than other frameworks.

Architecture

Beyond the GPU Abstraction

Developers often ask: "If you're using gogpu, what did you actually build?"

gogpu provides a low-level bridge to Vulkan and Metal. It's essentially a blank canvas. To make it a professional **engine**, we built several core "++" layers that simply didn't exist before:

  • The "Zero-Jank" Task Queue: We built a specialized buffer that allows your background tasks (like downloading a 1GB file) to update the UI without the app "hiccupping." Standard GPU apps usually freeze for a split second; Gödel stays at a smooth 60fps.
  • Intelligent Redrawing: We added a system that knows exactly which pixels changed. If you only move a slider, we don't redraw the whole screen—we only update the slider. This is why our idle battery usage is effectively zero.
  • Hot-Reloading (godel dev): we built a CLI that watches your code and pushes updates to the running app in under 2 seconds. This developer experience was previously almost impossible in native Go GPU projects.
  • Retained Mode Tree: A managed widget tree that tracks state changes, so you don't have to redraw everything manually every frame.

Typography

Rendering the Perfect Glyph

How do we make text look so crisp on an 8K display while using almost zero memory? The answer lies in our Multi-Channel Signed Distance Field (MSDF) font engine.

For the Layman: Why it’s so sharp

Standard fonts are often drawn using "rasterization"—turning lines into a grid of dots. If you scale them up, they get blurry. If you uses high-res images of every font size, your app becomes huge.

Gödel treats fonts like vectors. We store a small "guide map" (an SDF) for each letter. When the GPU draws a letter, it looks at the map and says "I am 2 pixels away from the edge," and draws a perfectly sharp line. Result: Perfect text at any size, with a tiny app footprint.

For the Expert: Multi-Channel (MSDF)

Standard SDFs have a weakness: they round off sharp corners because they only store a single distance value. Gödel utilizes MSDF, which stores orthogonal signals across the R, G, and B channels.

In our fragment shader, we sample these channels and calculate the median. This preserves the sharp intersection of edges, allowing for perfect glyph reconstruction even at low sampling frequencies. We then apply sub-pixel anti-aliasing via smoothstep(-w, w, dist) where w is the screen-space derivative of the distance.

Rendering Core

Vector Graphics: Our SDF Contribution

One of Gödel's biggest departures from standard rendering frameworks is how we handle shapes and text using Signed Distance Fields (SDF).

For the Layman: Infinite Zoom

Most apps use "bitmaps" (images made of tiny dots) for icons and text. If you zoom in too much, they get blurry or "pixelated." Gödel is different. Instead of dots, we store mathematical descriptions of the shapes. This means our icons and text are "infinitely sharp"—no matter if you're on a 1080p monitor or an 8K display, the edges stay perfectly crisp. It also makes our app binary tiny because math takes up much less space than high-res images.

For the Expert: MSDF & fwidth()

Standard SDFs often struggle with sharp corners at low resolutions. We contributed a robust Multi-Channel SDF (MSDF) pipeline. By sampling orthogonal distances across three channels (R, G, B) and evaluating the median on the GPU, we preserve sharp corners even in complex typography.

// Anti-aliasing via screen-space derivatives
float d = median(texture(sdf_atlas, uv).rgb);
float alpha = smoothstep(-fwidth(d), fwidth(d), d);

Concurrency

Dealing with OS Realities

Desktop OS graphics APIs (like Cocoa or WinAPI) are notoriously single-threaded. If you mutate a widget from a background goroutine, the app will crash or flicker.

We solved this with a 1024-slot task buffer. Instead of complex locks, you send a closure. Gödel drains this queue at the start of every frame, ensuring your data is applied safely on the main thread.

The Grounded Benefit

For you, this means you can fire off 100 concurrent API requests and simple call app.QueueCallback() for each result. The engine handles the synchronization math so you don't have to. No deadlocks, no race conditions.

Roadmap

Limitations & Challenges

We are humble about where Gödel is today. While our core architecture is fast, we are still a young project with several hurdles to clear:

Where we are still growing
  • Ecosystem: Our library of complex widgets (like Rich Text Editors) is still being built out.
  • Accessibility: Native screen-reader support is currently in beta.
  • 3D Graphics: We are optimized for 2D UI; complex 3D rendering is not our current focus.

Our goal isn't to be everything to everyone—it's to be the best at high-performance, deterministic 2D desktop applications in Go.

Quick Start

Start Building

We've tried to make it as easy as possible to get started. Note the CGO_ENABLED flag for macOS users, as we bind to native Metal.

Terminal
# Install the CLI
go install github.com/intercode/godel/cmd/godel@latest

# Initialize and run
godel init my-app
cd my-app

# On macOS: export CGO_ENABLED=1
godel run
